# shop_analytics_level3 - dbt Project Overview

## Overview

This dbt project is the transformation layer of the **Data Engineer Level 3** pipeline.

The full architecture is:

**MySQL `source_shop` -> Python ingestion -> BigQuery `raw` -> dbt `staging` -> dbt `marts`**

The Python ingestion layer loads raw operational tables from MySQL into BigQuery using full refresh and `WRITE_TRUNCATE`.

The dbt project is responsible for:

- declaring the raw tables as dbt sources
- creating the staging layer in BigQuery
- creating the marts layer in BigQuery
- documenting models and sources
- validating data quality with dbt tests

The final warehouse structure is:

- **raw**: ingested source tables from MySQL
- **staging**: cleaned and standardized views
- **marts**: final business tables for analytics

---

## Purpose of the dbt Folder

The dbt folder contains all the transformation logic of the warehouse project.

It does **not** perform ingestion from MySQL.  
That part is handled by Python.

This folder is used to:

1. connect dbt to BigQuery
2. define raw sources
3. transform raw data into staging models
4. transform staging models into marts
5. apply tests and documentation
6. manage the transformation workflow in a structured and reproducible way

In practical terms:

- Python updates **raw**
- dbt reads **raw**
- dbt builds **staging**
- dbt builds **marts**

---

## Folder Structure

### `dbt_project.yml`
This is the main configuration file of the dbt project.

It defines:

- the dbt project name
- the dbt profile name to use
- the main project paths
- the model configuration
- how staging and marts are materialized

In this project:

- `staging` models are materialized as **views**
- `marts` models are materialized as **tables**

---

### `macros/`
This folder contains reusable dbt macros.

In this project it is used to override dbt’s default schema naming behavior, so that the final BigQuery datasets are created exactly as intended:

- `staging`
- `marts`

without unwanted composed names such as `staging_staging` or `staging_marts`.

---

### `models/`
This is the core folder of the dbt project.

It contains all transformation logic and metadata for the warehouse models.

It is organized into three main logical layers:

#### `models/sources/`
Contains source definition files.

These files tell dbt which raw BigQuery tables exist and how to reference them as sources.

This layer does not transform data.  
It only declares the raw input tables and can also include source-level tests.

#### `models/staging/`
Contains staging models and their metadata.

This layer is the first transformation layer after raw.

Its purpose is to:

- clean the source data
- cast data types
- standardize naming
- expose consistent columns for downstream models

These models are typically materialized as **views**.

#### `models/marts/`
Contains final analytical models and their metadata.

This layer is the business-facing layer.

Its purpose is to:

- build dimensions
- build fact tables
- expose clean analytical tables for querying and reporting

These models are materialized as **tables**.

---

### `seeds/`
This folder is used by dbt for static CSV-based tables.

It is part of the standard dbt structure, but in this project it is currently **not used**.

---

### `snapshots/`
This folder is used for dbt snapshots.

Snapshots are useful for historical tracking of changing records over time.

It is part of the standard dbt structure, but in this project it is currently **not used**.

---

### `tests/`
This folder is used for custom SQL tests.

In this project, most tests are currently defined inside `.yml` files rather than custom SQL test files, so this folder is currently **not central** to the implementation.

---

### `analyses/`
This folder is used for ad hoc SQL analysis files that are not materialized as models.

It is part of the standard dbt structure, but in this project it is currently **not used as a main layer**.

---

### `target/`
This is an autogenerated folder produced by dbt execution.

It contains:

- compiled SQL
- run artifacts
- manifest files
- execution output

It is useful for debugging and inspection, but it is **not** a place where business logic is written manually.

---

### `logs/`
This folder contains dbt execution logs.

It is useful for:

- debugging
- checking failures
- reviewing execution details

---

## Difference Between `.sql` and `.yml` Files

Inside the `models` folder, the project uses two main file types:

### `.sql` files
These files contain the actual transformation logic.

Each SQL file defines a dbt model.

Examples:

- reading from a raw source
- renaming columns
- casting fields
- joining staging models
- building dimensions and facts

In simple terms:

- `.sql` files **build the data**

### `.yml` files
These files describe and validate the resources.

They are used to define:

- sources
- model descriptions
- column descriptions
- tests
- metadata

In simple terms:

- `.yml` files **document and test the data**

## Ordered Overview of the Operations Performed

The project was built in the following logical sequence.

### Step 1 - BigQuery environment preparation
The BigQuery datasets were created:

- `raw`
- `staging`
- `marts`

---

### Step 2 - Python ingestion
A separate Python ingestion process was used to load MySQL tables into BigQuery raw tables.

The ingestion process:

- reads data from MySQL `source_shop`
- loads data into BigQuery `raw`
- uses full refresh
- uses `WRITE_TRUNCATE`
- handles logging and table-level error handling

---

### Step 3 - dbt profile setup
The dbt profile was configured to connect the local dbt project to BigQuery using a service account key.

Purpose:

- allow dbt to authenticate against Google Cloud
- connect the project to the `staging` default dataset
- enable execution of dbt commands

---

### Step 4 - dbt project configuration
The `dbt_project.yml` file was configured.

Purpose:

- define the dbt project behavior
- assign `view` materialization to staging
- assign `table` materialization to marts

---

### Step 5 - custom schema naming macro
The `generate_schema_name` macro was created.

Purpose:

- ensure the correct final dataset names in BigQuery
- avoid incorrect schema composition

---

### Step 6 - source layer definition
The `src_shop.yml` file was created.

Purpose:

- register the raw BigQuery tables as dbt sources
- make the raw layer available to staging models

---

### Step 7 - staging layer creation
The staging SQL models were created.

Purpose:

- clean and standardize raw data
- expose reusable upstream views for marts

---

### Step 8 - marts layer creation
The marts SQL models were created.

Purpose:

- build final analytical tables
- create dimensions and fact table

---

### Step 9 - tests definition
YAML test files were added for sources, staging, and marts.

Purpose:

- validate structural correctness
- validate keys
- validate relationships
- improve trust in the warehouse models

---

## Final Result

At the end of the process, the dbt project provides:

- a declared raw source layer
- a cleaned staging layer
- a final marts layer
- documentation and tests around the warehouse models

The project now supports a structured analytics workflow where:

- ingestion is handled in Python
- transformation is handled in dbt
- storage and execution are handled in BigQuery

This creates a clean Level 3 warehouse architecture that is closer to a real production-style modern data stack.
